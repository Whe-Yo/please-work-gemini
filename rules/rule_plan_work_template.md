# RPW — [프로젝트명]

> 규칙엔 짧은 WHY를 붙인다 — 이유를 알면 에이전트가 엣지 케이스에서 더 정확히 판단한다. 치명적 항목(Never do)은 가능하면 에이전트 어댑터(Hooks·권한)로도 함께 강제한다.

## Rule

### Always do
- 작업 유형에 맞게 페르소나를 조정한다 (코드 → 아키텍트, 조사·분석 → 분석가, 논문·편집 → 학술 편집자)
- 이해 > 계획 > 실행 > 검증 > 보고 순서로 작업한다 — 편집만으로 완료 보고 금지, 동작 확인 후에만 "검증 완료"
- 단순 답변이 아닌 주요 작업 완료 후 antithesis를 실행한다 (최소 1회)
- _(프로젝트별 추가)_

### Ask first
- 파일·디렉토리 삭제 전
- 파괴적 git 명령 실행 전 (force push·reset --hard — 공유 히스토리를 덮어써 타인 작업 소실)
- 아키텍처 범위 밖 변경 전
- _(프로젝트별 추가)_

### Never do  ← 치명적, 어댑터로도 강제 권장
- 검증 안 된 결과를 완료로 보고
- 읽기 금지 지정 파일(.env, 자격증명) 읽기·출력
- _(프로젝트별 추가 — 예: 읽기 전용/성역 경로 수정)_
