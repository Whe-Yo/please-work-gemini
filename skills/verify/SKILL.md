# Verify — 변경 검증

"코드를 고쳤다"와 "동작한다"는 다르다. 실행 결과를 관찰한 뒤에만 완료를 보고한다.

## 절차

1. **검증 수단 선택** (우선순위 순):
   - 해당 기능을 다루는 기존 테스트가 있으면 그것을 실행
   - 없으면 변경된 코드 경로를 실제로 타는 최소 실행 (스크립트 실행, import 확인, 컴파일)
   - 임시 검증 코드가 필요하면 `tmp/`에 만들고, 검증 후 정리
2. **실행 전 기준 정의**: "무엇이 나오면 성공인지"를 실행 전에 한 줄로 적는다.
3. **실행 및 관찰**: 명령을 실행하고 실제 출력을 확인한다. exit code 0만으로 성공 판정하지 않는다 — 기대한 출력이 실제로 나왔는지 본다.
4. **보고**:
   - 성공: 실행한 명령과 핵심 출력을 함께 보고.
   - 실패: 출력 그대로 + 원인 분석. 낙관적 포장 금지.
   - 검증 불가(환경 문제, 데이터 없음 등): "수정했으나 미검증"으로 명시하고 사용자가 검증할 수 있는 명령을 제시.

## 규칙
- 학습/렌더링처럼 오래 걸리는 작업은 전체를 돌리지 말고 1 iteration, 소량 데이터로 축소 실행해 경로만 확인한다.
- 글로벌 환경(base, global 등) 사용 금지 — 프로젝트 지정 환경에서 실행.
- 검증 중 발견한 기존 버그(내 변경과 무관)는 고치지 말고 보고만 한다.
